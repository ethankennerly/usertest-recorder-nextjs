what is a simple way to record camera from a web page using javascript or nextjs?
March 28, 2026
The simplest way to record camera video in a web page is using the MediaStream Recording API. This involves capturing the stream from the camera using navigator.mediaDevices.getUserMedia() and passing it to a MediaRecorder object to handle the recording chunks. 
MDN Web Docs
MDN Web Docs
 +2
1. Basic Implementation (Vanilla JavaScript)
The recording process follows three main steps: access the device, start the recorder, and collect data chunks into a "Blob" for saving or playback. 
addpipe.com
addpipe.com
 +1
javascript
let mediaRecorder;
let recordedChunks = [];

async function startRecording() {
  // 1. Get access to camera and microphone
  const stream = await navigator.mediaDevices.getUserMedia({ video: true, audio: true });
  
  // 2. Initialize MediaRecorder with the stream
  mediaRecorder = new MediaRecorder(stream);
  
  // 3. Collect data as it becomes available
  mediaRecorder.ondataavailable = (event) => {
    if (event.data.size > 0) recordedChunks.push(event.data);
  };

  // 4. Handle what happens when recording stops
  mediaRecorder.onstop = () => {
    const blob = new Blob(recordedChunks, { type: 'video/webm' });
    const url = URL.createObjectURL(blob);
    // You can now set this URL to a <video src="..."> or download it
  };

  mediaRecorder.start();
}

function stopRecording() {
  mediaRecorder.stop();
}
Use code with caution.

2. Next.js / React Considerations
When implementing this in Next.js, you must ensure the code runs only on the client side, as navigator and mediaDevices are not available during server-side rendering (SSR).

The simplest approach for cross-platform (laptop and mobile) video and audio recording is using the MediaRecorder API in a client-side Next.js component. This allows you to capture the participant's reaction while they play your Unity WebGL game.

To handle Amazon S3 storage without exposing secret keys in the browser, use a single Next.js API route to generate a Presigned URL. This allows the browser to upload the video directly to S3 without sending the heavy video file through your server first.
1. Client-Side Recorder (Next.js Component) 
This component handles the "Start/Stop" logic and triggers the camera and microphone permissions.
javascript
import { useState, useRef } from 'react';

export default function CameraRecorder({ sessionId }) {
  const [recording, setRecording] = useState(false);
  const mediaRecorderRef = useRef(null);
  const chunksRef = useRef([]);

  const startRecording = async () => {
    const stream = await navigator.mediaDevices.getUserMedia({ 
      video: { facingMode: "user" }, // Selfie cam for mobile/laptop
      audio: true 
    });
    
    mediaRecorderRef.current = new MediaRecorder(stream, { mimeType: 'video/webm' });
    chunksRef.current = [];

    mediaRecorderRef.current.ondataavailable = (e) => {
      if (e.data.size > 0) chunksRef.current.push(e.data);
    };

    mediaRecorderRef.current.onstop = async () => {
      const blob = new Blob(chunksRef.current, { type: 'video/webm' });
      const { url } = await fetch('/api/get-s3-url', {
        method: 'POST',
        body: JSON.stringify({ filename: `${sessionId}.webm` }),
      }).then(res => res.json());

      await fetch(url, { method: 'PUT', body: blob, headers: { 'Content-Type': 'video/webm' } });
    };

    mediaRecorderRef.current.start();
    setRecording(true);
  };

  return (
    <button onClick={recording ? () => mediaRecorderRef.current.stop() : startRecording}>
      {recording ? "Stop & Upload" : "Start Record"}
    </button>
  );
}
Use code with caution.

2. Backend API Route (Presigned URL)
Create this file at /api/get-s3-url.js to securely talk to AWS. Use the AWS SDK for JavaScript to generate the upload link.
javascript
import { S3Client, PutObjectCommand } from "@aws-sdk/client-s3";
import { getSignedUrl } from "@aws-sdk/s3-request-presigner";

export default async function handler(req, res) {
  const s3 = new S3Client({ region: process.env.AWS_REGION });
  const { filename } = JSON.parse(req.body);

  const command = new PutObjectCommand({
    Bucket: process.env.S3_BUCKET_NAME,
    Key: `recordings/${filename}`,
    ContentType: "video/webm",
  });

  const url = await getSignedUrl(s3, command, { expiresIn: 3600 });
  res.status(200).json({ url });
}
Use code with caution.

Key Considerations for 2026
Mobile Support: Ensure you use facingMode: "user" in your getUserMedia constraints to default to the front camera on smartphones.
Unity Overlay: Since your Unity game uses a <canvas> element, place this React component in a fixed or absolute container with a high z-index so it sits on top of the game.
Browser Permissions: Both iOS and Android browsers require a user gesture (like a button click) to start recording or play audio.

To handle a user quitting (closing the tab, navigating away, or clicking an "Exit" button), you need a strategy that handles both graceful exits and abrupt crashes.
1. The "Graceful Exit" (Best UX)
The most reliable way is to have a "Finish Session" button in your Next.js UI (outside the Unity canvas).
When clicked, it calls mediaRecorder.stop().
The onstop event triggers your S3 upload.
Show a loading spinner: "Saving your session... do not close the tab."
2. The "Abrupt Exit" (Tab Closed)
If a user simply closes the browser tab, the MediaRecorder stops immediately, but the onstop callback usually won't have time to finish a multi-megabyte upload to S3.
To solve this, use the beforeunload event to warn the user, and the onpagehide event for the safest cleanup.
javascript
// Add this inside your useEffect in Next.js

  const handleBeforeUnload = (e) => {
    if (isRecording) {
      // Standard browser "Are you sure you want to leave?" popup
      e.preventDefault();
      e.returnValue = ''; 
    }
  };

  window.addEventListener('beforeunload', handleBeforeUnload);
  return () => window.removeEventListener('beforeunload', handleBeforeUnload);
}, [isRecording]);

3. The "Fail-Safe" (Local Storage)
For the highest reliability, save the video chunks to IndexedDB (via a library like dexie or localForage) as they are recorded.
How it works: Every time ondataavailable fires, you save that chunk to the user's browser storage.
The Benefit: If the tab crashes or they lose internet, the video is still on their laptop/phone.
Recovery: The next time they visit your site, your code checks: "Is there an unsent video in IndexedDB?" If yes, it uploads it to S3 in the background.

can videos be saved in timed chunks and uploaded in chunks? is there a simple way to stitch videos?
To avoid losing a long session if the user quits abruptly, you can record and upload in timed chunks (e.g., every 30 seconds). This ensures that if they close the tab at minute 10, you still have the first 9.5 minutes safely on S3.
1. Recording in Chunks (Frontend)
The MediaRecorder.start(ms) method accepts a timeslice in milliseconds. This triggers the ondataavailable event repeatedly at that interval.
javascript
const startRecording = async () => {
  const stream = await navigator.mediaDevices.getUserMedia({ video: true, audio: true });
  mediaRecorderRef.current = new MediaRecorder(stream);

  mediaRecorderRef.current.ondataavailable = async (e) => {
    if (e.data.size > 0) {
      // 1. Get a unique presigned URL for THIS chunk
      const chunkId = Date.now();
      const { url } = await fetch('/api/get-s3-url', {
        method: 'POST',
        body: JSON.stringify({ filename: `${sessionId}_${chunkId}.webm` }),
      }).then(res => res.json());

      // 2. Upload this specific chunk immediately
      await fetch(url, { method: 'PUT', body: e.data });
    }
  };

  // Trigger 'ondataavailable' every 30 seconds (30000ms)
  mediaRecorderRef.current.start(30000); 
};
Use code with caution.

2. Stitching Videos (Simplest Way)
Stitching videos in a browser is complex because of encoding overhead. The most "frictionless" ways to handle this are:
Option A: Stitch on Playback (Zero Processing Cost)
Instead of merging the files into one giant video, keep them as separate chunks in S3.
The Logic: Store a list of chunk URLs in your database (or PostHog metadata).
The Player: Use a simple video player logic that plays chunk_1.webm, then automatically switches the src to chunk_2.webm when the first one ends.
Option B: FFmpeg on the Server (Next.js)
If you truly need a single file, use FFmpeg. Since you are on Vercel, you shouldn't run heavy video processing in a Serverless Function (it will timeout).
The Tool: Use a tool like Cloudinary or Mux. You send them the S3 URLs, and they merge them via their API for a few cents.
Self-Hosted: If you have a separate backend (like DigitalOcean or AWS EC2), you can run a simple FFmpeg command:
ffmpeg -f concat -safe 0 -i file_list.txt -c copy output.webm
Option C: Client-Side Stitching (Advanced)
You can use ffmpeg.wasm to merge files directly in the user's browser before the final upload, but this is heavy and might crash a mobile browser or a laptop already struggling to run a Unity game.
Recommendation for a Playtest Tool
Go with Option A (Stitch on Playback). It is the most robust because:
No Data Loss: If an upload fails, only 30 seconds are lost.
Instant Availability: You don't have to wait for a "rendering" or "merging" step to finish after the user quits.
Cost: It costs $0 in processing fees.

To achieve a "streamed" feel with high reliability and zero frame rate hitches, 1-second slices are actually counter-productive. They create massive overhead (CPU and Network) due to the sheer number of HTTP requests and SSL handshakes.
For a Unity game—which is already competing for the CPU’s main thread—the most efficient "sweet spot" is a 30-second slice or a continuous stream.
1. Why 1-Second Slices Cause "Hitches"
Network Overhead: Every 1 second, your browser has to perform a POST or PUT request. This involves DNS lookups, TCP handshakes, and SSL/TLS negotiation. On mobile, this will quickly overheat the device and cause the game's frame rate to stutter.
Disk/Memory I/O: Constantly cutting and closing video containers (WebM headers) every second is more CPU-intensive than letting the encoder run in a steady state.
2. The Efficient "Golden Middle": 10–30 Second Slices
If you want to ensure you don't lose data if the user quits, use a 10 to 30-second interval.
CPU: The MediaRecorder runs on a separate browser thread. By giving it a larger "timeslice" (e.g., mediaRecorder.start(30000)), the browser can optimize the encoding buffer.
Bandwidth: You send one larger file instead of 30 tiny ones, which is much more efficient for S3 and the user's data plan.
3. The "True" Ideal: S3 Multipart Upload (Streamed Transfer)
If you want to "stream" the video to S3 as it happens without individual files, you should use the S3 Multipart Upload API. This allows you to:
Open a "Upload Session" when the game starts.
Send "Parts" (chunks) as they are recorded.
"Complete" the upload when the user quits.
The Workflow:
Frontend: MediaRecorder captures 5MB chunks (the minimum size AWS allows for multipart parts).
Next.js API: Forwards these chunks to S3.
Result: You get one single video file in S3 at the end, but the data was uploaded progressively during the game.
4. Code Strategy for Zero Hitching
To ensure the game stays smooth, use a Web Worker for the upload logic. This offloads the network "wait time" to a background thread so it doesn't block the Unity game's rendering.
